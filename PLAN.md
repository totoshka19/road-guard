# RoadGuard — real-time дашборд мониторинга дорожных нарушений

> **Название:** `RoadGuard` — так пишем в UI (шапка дашборда, `<title>`), в README и в текстах для портфолио. В `package.json` (поле `name`) и в имени папки/репозитория — kebab-case: `road-guard`.

---

## О проекте

Веб-дашборд, который в реальном времени показывает на карте города дорожные камеры фотовидеофиксации и «прилетающие» с них нарушения ПДД. Тема сознательно выбрана под домен компании (безопасность дорожного движения, фотовидеофиксация, GIS, компьютерное зрение).

**Главный принцип реализации:** для посетителя всё выглядит как настоящий real-time, но данные генерируются в браузере (без бэкенда). Архитектура специально спроектирована так, чтобы позже подменить симуляцию на реальный сервер (`fetch` + `WebSocket`) без переписывания UI. Это даёт бесплатный хостинг статики и мгновенную загрузку по ссылке.

## Цель

Портфолио-проект под вакансию frontend-разработчика (React). Демонстрирует и подтверждает навыки:

- **Ядро вакансии:** React + TypeScript, работа с REST API, ООП (интерфейсы/абстракции), Git, понимание CI/CD.
- **Пункты «будет преимуществом»:** GIS-системы, real-time UX, обработка WebSocket, PWA, unit-тесты.
- Публичная ссылка на бесплатном хостинге — можно приложить к отклику.

## Целевой результат

Задеплоенное на бесплатном хостинге SPA с живой ссылкой + чистый репозиторий с осмысленной историей коммитов и README со скриншотами/gif.

## Технический стек

Бóльшая часть выбрана из уже знакомых технологий, чтобы проект подтверждал существующие навыки и добавлял новые.

### Ядро
- **Vite** — сборщик, быстрый HMR, из коробки собирает статику для деплоя.
- **React 19 + TypeScript** (strict mode) — прямое требование вакансии.
- **React Router** — роутинг: главный дашборд + страница отдельной камеры. _(Фактически подключён v8 — на момент реализации это актуальный мажор.)_

### Состояние и данные
- **Redux Toolkit** — управление состоянием (стандарт индустрии).
- **RTK Query** — «REST»-загрузка начальных данных (список камер, история нарушений); идиоматично демонстрирует работу с REST API.
- **reselect** (`createSelector`, входит в RTK) — производная статистика без лишних ре-рендеров.

### Карта / GIS (главный showcase)
- **MapLibre GL JS** + **react-map-gl** — open-source, вектор на WebGL, **нативные** кластеризация маркеров и heatmap-слой, работает **без API-ключа**.
- Тайлы: **OpenFreeMap** (`https://tiles.openfreemap.org/styles/dark`) или demotiles MapLibre — бесплатно, без ключа.
- _Более простая альтернатива при желании:_ Leaflet + react-leaflet + плагины `leaflet.heat` / `leaflet.markercluster`.

### Графики
- **Chart.js** + **react-chartjs-2** — doughnut (по типам), line (динамика по времени), bar (по районам). _Альтернатива: Recharts._

### Стилизация и UI
- **Tailwind CSS** — быстро для дашборда, современный вид. _Альтернатива: SCSS + БЭМ, если хочешь явно показать методологию._
- **lucide-react** — иконки.
- _(опционально)_ **shadcn/ui** — готовые компоненты на Tailwind для ускорения.

### Анимации, типы, утилиты
- **GSAP** — пульс появляющихся нарушений на карте, анимированные счётчики KPI, выезд панелей. _Альтернатива: Framer Motion._
- **Zod** — схемы и валидация формы мок-данных и входящих событий → строгая типизация.
- **date-fns** — форматирование времени в ленте, фильтры по времени.
- **@faker-js/faker** — генерация правдоподобных данных (номера, ID, время).

### PWA
- **vite-plugin-pwa** — устанавливаемое приложение, service worker, офлайн-оболочка.

### Качество и CI
- **ESLint + Prettier** — качество кода.
- **Vitest + React Testing Library** — unit- и компонентные тесты.
- **GitHub Actions** — CI (lint + тесты + сборка) и автодеплой; прямой аналог GitLab CI/CD.

### Хостинг
- **Cloudflare Pages** или **Vercel** — бесплатно, SPA-роутинг из коробки.

### Окружение
- Node.js 20+; менеджер пакетов — npm или pnpm (на выбор).

## Архитектура

Сердце проекта — слой абстракции данных, отделяющий симуляцию от остального кода.

```
src/
  app/            # store.ts, типизированные хуки (useAppDispatch/useAppSelector)
  data/           # ← слой абстракции данных
    types.ts      #   TS-типы + Zod-схемы (Camera, Violation, ViolationType)
    DataSource.ts #   интерфейс источника данных
    mock/
      MockDataSource.ts     # реализация «сейчас» (симуляция)
      violationGenerator.ts # генератор потока событий
      city.ts               # bbox города, районы, типы нарушений
    real/         #   пустой задел под будущую реализацию (WS + fetch)
  features/
    cameras/      # слайс, селекторы, маркеры камер
    violations/   # слайс (кольцевой буфер ~500), лента, маркеры нарушений
    stats/        # производные селекторы + графики
    filters/      # слайс + UI фильтров
    playback/     # управление стримом (пауза, скорость)
  components/     # общий UI (Map, Card, Panel, KpiCard…)
  pages/          # DashboardPage, CameraDetailPage
  lib/            # утилиты (гео, форматирование)
  styles/
  main.tsx
```

### Ключевые решения

**1. Единый интерфейс `DataSource`.** Весь UI работает через него и не знает, откуда приходят данные:

```ts
// data/DataSource.ts
export interface DataSource {
  getCameras(): Promise<Camera[]>;
  getViolations(filters: ViolationFilters): Promise<Violation[]>;
  /** Подписка на поток новых нарушений. Возвращает функцию отписки. */
  subscribe(onViolation: (v: Violation) => void): () => void;
}
```

Сейчас работает `MockDataSource`. Позже создаётся `RealDataSource` с настоящими `fetch` + `WebSocket` — остальной код не меняется. Это и есть тезис для собеседования: «подключить реальный бэкенд — тривиально».

**2. WS-подобный мок.** Генератор внутри имитирует поведение WebSocket-клиента (`connect` / `subscribe` / `unsubscribe`) и эмитит новое нарушение каждые N мс с джиттером и всплесками, чтобы поток выглядел живо. Код обработки real-time пишется ровно такой же, как для настоящего сервера.

**3. Кольцевой буфер нарушений.** Храним последние ~500 событий, старые вытесняем — иначе за час демо стор распухнет. Статистика считается селекторами поверх буфера.

### Доменная модель

```ts
// data/types.ts (эскиз)
export type ViolationType =
  | 'speeding'          // превышение скорости
  | 'red_light'         // проезд на красный
  | 'wrong_way'         // выезд на встречную
  | 'no_pedestrian'     // непропуск пешехода
  | 'illegal_parking'   // парковка в неположенном месте
  | 'shoulder_driving'; // движение по обочине

export interface Camera {
  id: string;
  name: string;
  lat: number;
  lng: number;
  district: string;
}

export interface Violation {
  id: string;
  cameraId: string;
  type: ViolationType;
  plate: string;      // госномер
  lat: number;
  lng: number;
  district: string;
  timestamp: number;  // epoch ms
}
```

Такая конкретика делает демо «своим» под бизнес компании.

## Функционал

- **Карта** во весь экран: маркеры камер с кластеризацией; новые нарушения появляются точками с анимацией (GSAP).
- **Лента событий** сбоку: прокручивающийся список входящих нарушений (тип, адрес, время, камера).
- **Фильтры:** по типу нарушения, временнóму окну, району.
- **Панель статистики:** KPI-карточки (всего за день, активных камер, разбивка по типам) + три графика + переключатель **тепловой карты** плотности нарушений.
- **Контролы плейбека:** пауза/возобновление потока, скорость 1x / 2x / 5x.
- **Страница камеры:** инфо о выбранной камере + её последние нарушения + мини-график.
- **Полировка:** тёмная/светлая тема, адаптив под мобильный, состояния загрузки/пустоты.

→ **Результат:** пустой, но живой по ссылке дашборд. **Закрывает:** Git, CI/CD.

Типы + Zod-схемы, интерфейс `DataSource`, `MockDataSource` с генератором и WS-подобным стримом, Redux-стор со слайсами `cameras`/`violations` и кольцевым буфером. UI ещё нет — проверяем в консоли и первых тестах.

### Этап 2 — карта с камерами (~1–2 дня)

## Хостинг и деплой

Собранная статика (`vite build` → `dist/`) деплоится бесплатно на **Cloudflare Pages** или **Vercel** — оба дают SPA-fallback из коробки и не «засыпают». GitHub Actions собирает и публикует на каждый push в `main`.

## Как подключить реальный бэкенд позже

1. Реализовать `RealDataSource` в `data/real/`: `getCameras`/`getViolations` через `fetch` (REST), `subscribe` через `WebSocket`.
2. В `store.ts` подменить `MockDataSource` на `RealDataSource` — точка подмены одна.
3. UI, слайсы и селекторы не трогаются.

## Соглашения

- **TypeScript strict.** Файлы компонентов — `PascalCase.tsx`, остальное — `camelCase.ts`.
- **ESLint + Prettier** прогоняются в CI; коммит не должен ломать lint/тесты/сборку.
- **Коммиты** — Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`).
- **Ветки:** работа в feature-ветках, merge в `main` через PR (даже соло — ради чистой истории и демонстрации процесса).
